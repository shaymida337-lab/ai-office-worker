# מדריך Rollout לפרודקשן — הקשחת צינור הסריקה

> נוצר בשלב 7 של פרויקט ההקשחה (2026-07-04). מכסה את commits
> `f061658`..`dd798e9` + commit המסמכים של שלב 7.
> **כלל ברזל: לא מדלגים על צעד, ועוצרים בכל נקודת "עצור" עד שהתנאי מתקיים.**

---

## צעד 1 — Push

```bash
git push origin main
```

- **מה לצפות:** דחיפה נקייה של ~8 commits (שלבים 0-7).
- **עצור אם:** ה-push נדחה (rebase נדרש) — לא לדחוף בכוח.

## צעד 2 — Deploy (Render)

ה-deploy אוטומטי מ-main. לעקוב בדשבורד של Render.

- **מה לצפות:** build ירוק, service עולה. אין מיגרציות DB חדשות בשלבים
  האלה (אפס שינויי סכימה!) — לא אמור לרוץ `migrate deploy` על שינוי כלשהו.
- **עצור אם:** ה-build נכשל או ה-service לא עולה — rollback ב-Render וחקירה.

## צעד 3 — אימות עשן (5 דקות)

1. פתח את הדשבורד → טאב חשבוניות: ודא שהרשימות נטענות.
2. **הבדיקה המרכזית (F12):** טאב "מאושר" אמור להציג עכשיו גם רשומות
   שאושרו אוטומטית (auto_saved) — צפוי **גידול** במספר הפריטים בטאב.
   זה התיקון, לא רגרסיה.
3. הרץ סריקת Gmail ידנית קצרה (quick scan) וודא שהיא מסתיימת תקין.
4. סרוק חשבונית אחת דרך המצלמה — ודא שנוצר קובץ ב-Drive (חדש! עד עכשיו
   מסלול המצלמה לא העלה ל-Drive בכלל).
- **עצור אם:** רשימות לא נטענות / סריקה נכשלת — rollback.

## צעד 4 — Baseline פרודקשן אמיתי (read-only)

בסביבת production shell (או מקומית מול DB פרודקשן **לקריאה בלבד** —
הסקריפט מסרב לרוץ על host שאינו localhost, לכן בפרודקשן מריצים אותו
מהשרת עצמו או מסירים זמנית את המגבלה בעותק מקומי של הסקריפט):

```bash
cd backend && npx tsx scripts/scan-quality-report.ts > scan-quality-prod-$(date +%F).txt
```

- **מה לצפות:** המדדים האמיתיים לראשונה: ספקי junk, סכומי 0, תאריכים
  בעייתיים לפי מקור, כפילויות, קישורי Drive חסרים, פילוח סטטוסים,
  ופער early-NEEDS_REVIEW. **לשמור את הקובץ** — זה ה-baseline להשוואה.
- **עצור אם:** — (read-only, אין סיכון). אם המספרים מפתיעים בסדר גודל —
  לתעד לפני שממשיכים.

## צעד 5 — תיקון סטטוסים ישנים: dry-run

```bash
cd backend && npx tsx scripts/fix-review-status-consistency.ts
```

- **מה לצפות:** רשימת GSI שנדרסו בעבר ל-needs_review ע"י שגיאה מאוחרת
  למרות שהתשלום/החשבונית נשמרו. כל שורה: id, ספק, סכום, הודעה.
- **עצור כאן — חובה:** מעבר ידני על הרשימה. כל שורה חשודה (ספק מוזר,
  סכום לא הגיוני) — לא ממשיכים עד שמבינים אותה.

## צעד 6 — תיקון סטטוסים: הרצה בפועל

**רק אחרי גיבוי DB ומעבר ידני מלא על ה-dry-run:**

```bash
cd backend && npx tsx scripts/fix-review-status-consistency.ts --apply
```

- **מה לצפות:** אותן רשומות בדיוק מה-dry-run מסומנות UPDATED, עם
  decisionReason חדש `restored_after_late_failure`.
- **עצור אם:** מספר הרשומות שונה מה-dry-run — משהו השתנה בין הריצות.

## צעד 7 — כפילויות קיימות: דוח לניקוי ידני (read-only)

```bash
cd backend && npx tsx scripts/find-duplicates.ts > duplicates-$(date +%F).txt
```

- **מה לצפות:** קבוצות עם קוד-סיבה: `invoice_number_and_amount` (כמעט
  ודאי כפילות), `same_gmail_message` (סימפטום F6 ההיסטורי),
  `supplier_amount_same_day` (חשד בלבד — ייתכנו חיובים לגיטימיים זהים!).
- **הניקוי ידני בלבד:** הסקריפט לא מוחק. לכל קבוצה — להחליט אנושית.
  מומלץ להתחיל מ-`invoice_number_and_amount` בלבד.

## צעד 8 — רשומות שממתינות ל-Drive (read-only)

```bash
cd backend && npx tsx scripts/find-pending-drive.ts
```

- **מה לצפות:** רשומות עם `pending_retry` — ה-retry האוטומטי (כולל
  ה-retry החדש למצלמה) משלים אותן בהדרגה בכל סנכרון. אם backlog לא
  יורד בין ריצות — לחפש `DRIVE_RETRY_BACKLOG` בלוגים ולחקור.

## צעד 9 — דוח איכות חוזר (אחרי שבוע)

```bash
cd backend && npx tsx scripts/scan-quality-report.ts > scan-quality-prod-week1.txt
diff scan-quality-prod-<baseline>.txt scan-quality-prod-week1.txt
```

- **מה לצפות מול ה-baseline של צעד 4:**
  - ירידה בספקי junk ברשומות חדשות (F1)
  - אפס תאריכים out-of-±2y חדשים ב-whatsapp/camera (F4)
  - אפס רשומות camera חדשות בלי קישור Drive (שלב 5)
  - פילוח סטטוסים: אפס GSI חדשים במצב "לא מופיע בכלל" (F12)
  - כפילויות חדשות: 0 בקבוצות `invoice_number_and_amount`

## נקודות מעבר ידועות (לא תקלות)

1. **גל review חד-פעמי קטן:** fingerprints ישנים חושבו בלי תאריך (F16) —
   סריקות חוזרות של מסמכים ישנים עשויות לייצר רשומות review בודדות
   בימים הראשונים. ההגנות המשלימות (emailMessageId, close_dates) מגשרות.
2. **גידול בטאב "מאושר":** רשומות auto_saved שהיו בלתי-נראות מופיעות
   עכשיו — זה התיקון של F12, לא באג.
3. **סכומים דו-משמעיים ל-review:** מסמכים עם "110.723"-style שהיו
   מנוחשים בעבר מגיעים עכשיו לתור הבדיקה (F3) — עומס review קטן נוסף,
   מכוון.
